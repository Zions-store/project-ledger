# STYLE_GUIDE Markers

The auditor reads three machine-readable tables from `STYLE_GUIDE.md`. These
tables can be wrapped in **language-independent markers** so they parse correctly
regardless of the document's language.

## Marker format (recommended, generated STYLE files)

```markdown
<!-- AUDIT: ENABLED_DOCS_START -->
| File | Role |
|---|---|
| Design_Document.md | GDD |
| Gameplay_Systems.md | systems |
<!-- AUDIT: ENABLED_DOCS_END -->
```

The three marker keys are:

| Marker key | Contains |
|---|---|
| `ENABLED_DOCS` | The authoritative document list (one `.md` per row) |
| `ANCHOR_REGISTRY` | The anchor ID table (columns: Anchor ID / Setting / Authority / Ref) |
| `DEPRECATED_TERMS` | The deprecated-term table (columns: Deprecated / Correct now / Type / Keyword / Scope) |

The `STYLE_GUIDE_TEMPLATE.md` shipped with the Skill already includes these
markers. When a project is scaffolded or its STYLE is generated, the markers
are filled with the project's data.

## Fallback (legacy STYLE files)

If a STYLE file has **no markers**, the auditor falls back to heading heuristics:

- `## 2. 文件清单` or `## 2. Authoritative File List` → document list table
- `已建立锚点清单` or `anchor registry` → anchor table
- `废弃说法` + `当前正确说法` → deprecated-term table (stops at `登记原则`)

This fallback keeps existing projects auditable without modification, even if
their STYLE was written before the marker system was introduced.

## Best practice

Use markers in all new projects. They are language-independent (the auditor does
not parse the heading text) and are generated by `gdd-scaffold` automatically.
